# 集成测试开发指南

> 为 `ecmdb` 提供一套标准化的、高隔离度且易于维护的集成测试方案。

---

## 1. 核心目标
集成测试的目的是**验证业务全链路的真实数据流**。

### 1.1 “混合治理”测试哲学 (Mixed Architecture)
集成测试并非“全量真实”，而是在**真实基础设施**与**精确 Mock** 之间取得平衡：

*   **本模块内 (Intra-module) -> 保持真实**：
    *   **数据库**：涉及本模块业务表的 SQL/NoSQL 操作必须使用真实数据库。
    *   **消息队列**：涉及本模块内的事件流转应使用真实的内存 MQ 逻辑。
*   **跨模块依赖 (Inter-module) -> 建议 Mock**：
    *   当 `Event` 模块依赖 `Order` 或 `User` 的 Service 接口时，建议 Mock 这些接口。这可以避免测试套件过度臃肿，并能精准测试 `Event` 模块对不同依赖输出的响应。
*   **外部网关 (External Gateway) -> 强制 Mock**：
    *   涉及第三方 API（如飞书、企业微信、外部支付网关）的调用必须 Mock，以保证测试环境的独立性和稳定性。

---

## 2. 基础设施初始化 (Shared IOC)

所有的集成测试应统一通过 `internal/test/ioc` 包来获取基础设施实例，确保连接配置的一致性。

```go
func (s *MyTestSuite) SetupSuite() {
	s.mongo = ioc.InitMongoDB()
	s.db = ioc.InitMySQLDB()
}
```

> [!TIP]
> 内存 MQ (mq-api/memory) 是集成测试的最佳选择，它能模拟真实的生产消费流程而无需搭建复杂的中间件集群。

---

## 3. 依赖注入与 TestApp 模式

使用 `wire` 生成一套专门用于测试的 `TestApp`。在 `internal/xxx/integration/startup/wire.go` 中定义 `InitApp` 函数，允许在初始化时动态注入外部依赖的 Mock。

```go
// startup/wire.go
func InitApp(mockSender sender.NotificationSender, ...) (*TestApp, error) {
    wire.Build(
        ioc.BaseSet,          // 基础 DB/MQ 供应集
        service.ProviderSet,  // 业务逻辑供应集
        wire.Struct(new(TestApp), "*"),
    )
}
```

## 4. 测试套件生命周期 (Suite)

采用 `github.com/stretchr/testify/suite` 管理测试生命周期，确保用例之间的环境隔离。

### 4.1 严格的环境隔离
每个测试用例执行前后必须清理数据，避免数据污染。

```go
func (s *NotificationSuite) cleanup() {
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()
	// 使用 TRUNCATE (SQL) 或 DeleteMany (NoSQL)
	s.mongo.Collection("c_user").DeleteMany(ctx, map[string]interface{}{})
}

func (s *NotificationSuite) SetupTest() {
	s.cleanup()
}

func (s *NotificationSuite) TearDownTest() {
	s.cleanup()
}
```

## 5. Table-Driven Tests 进阶模式

在传统的 Table-Driven 模式基础上，引入 `before` 和 `after` 钩子，使测试代码结构化。

```go
func (s *NotificationSuite) TestMyService() {
	testCases := []struct {
		name   string
		before func(t *testing.T) // 准备数据库数据、设置 Mock 期待
		after  func(t *testing.T) // 深度校验数据库结果、验证 Mock 调用
		
		req     Request
		wantErr error
	}{
		{
			name: "成功场景",
			before: func(t *testing.T) {
				// 插入真实用户数据
			},
			after: func(t *testing.T) {
				// 验证数据库中某字段已更新
			},
		},
	}
    // ... 执行循环
}
```

## 6. 自定义断言助手

针对复杂的业务实体，编写特定的断言工具函数，自动处理 `ID`、`Ctime`、`Utime` 等动态字段。

```go
func (s *NotificationSuite) assertUserEqual(t *testing.T, expected, actual User) {
    actual.ID = expected.ID     // 忽略自增 ID
    actual.Ctime = expected.Ctime // 忽略时间戳差异
    require.Equal(t, expected, actual)
}
```

## 7. 最佳实践总结 (Dos & Don'ts)

| 场景 | 推荐做法 | 禁止做法 |
| :--- | :--- | :--- |
| **数据库** | 使用 `cmdb_e2e` 等专用测试库 | 在生产或通用开发库运行 |
| **消息队列** | 使用 `mq-api/memory` 验证生产消费逻辑 | 仅 Mock Producer 接口而不验证消费端 |
| **数据校验** | 查询数据库验证物理状态 | 仅验证 Service 函数的 Error 返回值 |
| **环境变量** | 使用 `startup` 统一配置热切换 | 在代码中硬编码 IP 或端口 |

> [!IMPORTANT]
> 所有的集成测试文件应放在独立的 `integration` 包中，并建议添加 `//go:build e2e` 标签。
